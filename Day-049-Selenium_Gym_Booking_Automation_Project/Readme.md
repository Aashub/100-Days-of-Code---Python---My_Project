# Day 49 - Selenium Gym Booking Automation Project

## Project Overview

This is a Gym Class Booking Automation project built using Selenium. The script automatically logs into a gym website, finds specific classes (like Spin Class at 6:00 PM and Yoga Class at 5:00 PM), books them if available, joins the waitlist if needed, retries booking in case of failure, and finally verifies all bookings in the “My Bookings” section.

The script is designed to handle network errors, already booked classes, and dynamic booking windows. It also keeps track of total bookings, waitlists, and verification results. This project was created using Selenium WebDriver, environment variables, retry logic, dynamic DOM traversal, and structured function-based design.

## How It Works

* **login():** This function handles the complete login process. It locates the email and password input fields using their IDs, clears any existing text, and enters credentials securely using environment variables. After submitting the login form, it waits briefly and checks the current URL to determine whether login was successful, If the URL remains on the login page, it returns 1 (login failed) and If redirected successfully, it prints "Login Successful!" and returns 0.
* **retry():** This function repeatedly calls the login() function until login succeeds or all retry attempts are exhausted. It loops from attempt 1 up to the specified retry count. If login() returns 0, the loop breaks immediately. If it keeps failing, it continues retrying until the successful login. this function ensures temporary network or loading failures do not break the script. Once login succeeds, the script moves forward to locate class cards.
* **find_ancestor_div_element():** This function identifies the parent container (day section) for each class card element. It receives all elements whose ID starts with class-card-, then moves one level up in the DOM using `XPath ".."` to access the parent `<div>` element. From this parent container, it extracts the day title (h2 tag) and the container ID. It filters only specific days (such as Monday or Tuesday) and stores both the Selenium day objects and their corresponding day/date titles. It returns these lists for further processing.
* **selecting_booking_day():** This function builds a structured dictionary containing all class details for each selected day like Monday & Tuesday. For each day container element, it collects all class `card IDs`, `class titles (h3 tags)`, and `class timings (p elements)`. These values are stored inside a dictionary under keys like `"class_card", "class_title", and "class_timing"`. The function returns this dictionary along with the corresponding day/date list, which is later used for booking logic.
* **retrying_booking_class():** This function ensures that booking is retried multiple times in case of of unstable network or temporary failures. First, it calls check_already_present_booking() to verify whether classes are already booked or processed or not. If not, it attempts to book classes using book_class() and retries up to a fixed number of attempts. It continues retrying until booking succeeds or the maximum attempts are reached. This improves reliability in unstable network conditions.
* **check_already_present_booking():** This function checks whether the target classes are already booked or waitlisted before attempting to book again. It loops through all classes in the booking details dictionary and matches only the target classes (Spin Class at 6:00 PM and Yoga Class at 5:00 PM). If the button text shows "Booked" or "Waitlisted", it increments the global counter to register the details of already booked classes and prints the status of already booked classes. This prevents unnecessary booking attempts.
* **book_class():** This function performs the actual class booking process. It loops through each selected day and checks every class to find the target classes (Spin Class at 6:00 PM or Yoga Class at 5:00 PM). If a match is found, it locates the corresponding button. If the button text is "Book Class", it clicks to book. & If it is "Join Waitlist", it clicks to join the waitlist. After clicking, it checks whether the action was successful using `check_button_text()` function. If successful, it updates the global counters variables of all booked classes and waitlist joined and prints confirmation that classes are booked and waitlist are joined. If booking fails, it returns 1 to trigger retry logic in `retrying_booking_class()` function. If all processing completes, it returns "classes_get_booked" which lets us to stop the while loop in `retrying_booking_class()` function.
* **check_button_text():** This helper function verifies whether a booking action was successful after clicking the button. If the button text changes to "Booked" or "Waitlisted", it returns 0 (success). Otherwise, it returns 1 (failure). This return value is used to control retry behavior in the booking process.
* **class_booked_summary():** This function prints a summary of the booking results. It calculates the total number of processed classes by adding booked classes, joined waitlists, and already booked or waitlisted classes. It then displays a structured summary showing all counts clearly. 
* **class_booked_summary():** This function prints a detailed list of newly processed bookings. It checks the class title and timing for target classes and prints whether each class was newly booked or waitlisted. It also stores the class titles in a global list for later verification.
* **check_my_booking():** This function verifies all bookings on the “My Bookings” page. It clicks the “My Bookings” link and checks whether the page shows an empty state. If bookings exist, it collects confirmed bookings and waitlisted bookings using CSS selectors. It extracts booked days and prints the total count. Finally, it calls `verify_my_classes()` to compare expected bookings with actual bookings shown on the page.
* **verify_my_classes():** This function verifies whether the classes the script attempted to book are actually visible in the “My Bookings” section. It first removes duplicate entries from the expected booking list. Then, it extracts class titles from both confirmed and waitlisted booking cards by locating their h3 elements. After collecting all visible class names, it compares each expected class with the displayed class titles. If a match is found, it prints a verification message. Finally, it calls `display_verification_result()` to compare the total expected bookings with the total bookings found on the page.
* **display_verification_result():** This function calculates the difference between expected bookings and actual bookings found on the page. If both counts match, it prints a success message. If there is a mismatch, it prints how many bookings are missing. This serves as the final validation step of the automation workflow.
